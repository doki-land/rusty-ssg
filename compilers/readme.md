# Rusty SSG 编译器架构规范

## 1. 项目结构概述

所有模板引擎编译器必须遵循以下统一的项目结构，确保代码组织的一致性和可维护性。

### 1.1 基础目录结构

```
compiler-name/
├── bin/
│   └── compiler-name.rs       # RIIR 版本的主入口
├── src/
│   ├── compiler/              # 编译器核心逻辑
│   │   ├── parser/            # 语法解析器（使用 oaks）
│   │   ├── renderer/          # 渲染器
│   │   └── mod.rs
│   ├── config/                # 配置处理
│   ├── watcher/               # 文件监听器（用于 dev、HMR 等功能）
│   ├── data/                  # 数据处理
│   ├── errors/                # 错误处理
│   │   └── mod.rs
│   ├── plugin/                # 插件系统
│   ├── tools/                 # 工具函数
│   │   ├── cmd/               # 命令行工具
│   │   └── mod.rs
│   ├── types/                 # 类型定义
│   └── lib.rs                 # 库入口
├── tests/                     # 测试代码
├── templates/                 # 模板文件
├── Cargo.toml                 # 依赖配置
└── readme.md                  # 项目说明
```

## 2. 核心架构规范

### 2.1 编译流程

1. **配置解析**：读取和解析项目配置文件
2. **内容扫描**：遍历项目目录，收集所有需要处理的文件
3. **语法解析**：使用 oaks 解析器解析模板文件
4. **数据处理**：处理和准备模板所需的数据
5. **模板渲染**：将数据应用到模板中
6. **输出生成**：生成最终的静态文件

### 2.2 统一抽象

所有模板引擎的抽象都放在 `nargo/compilers/nargo-template` 中，包括：

- 模板解析接口
- 渲染器接口
- 配置处理接口
- 插件系统接口

### 2.3 解析器规范

所有解析器必须使用 `oaks/examples` 提供的解析器，禁止使用第三方解析库如：

- pulldown-cmark
- handlebars
- liquid
- askama
- yaml-rust
- serde\_yaml
- toml

### 2.4 依赖管理

- 使用 pnpm workspace 管理依赖
- 优先使用 nargo 提供的组件
- 禁止引入重型非自研库

## 3. 各模块详细规范

### 3.1 bin 目录

- 每个编译器的 bin 目录下只保留一个主入口文件：`compiler-name.rs`
- 这是 RIIR 版本的主入口，负责命令行解析和调用编译器核心功能
- 禁止在 src 目录下重复实现相同功能

### 3.2 src 目录

#### 3.2.1 compiler 模块

- **parser**：使用 oaks 实现的语法解析器
- **renderer**：负责将解析后的模板渲染为最终输出
- **mod.rs**：导出编译器核心功能

#### 3.2.2 config 模块

- 处理项目配置文件的读取和解析
- 提供统一的配置接口

#### 3.2.3 watcher 模块

- 实现文件系统监听器
- 支持开发模式（dev）和热模块替换（HMR）功能
- 监控文件变化并触发相应的编译和更新

#### 3.2.4 data 模块

- 处理模板所需的数据
- 支持从各种数据源获取数据

#### 3.2.5 errors 模块

- 定义错误类型和错误处理机制
- 提供统一的错误处理接口

#### 3.2.6 plugin 模块

- 插件系统实现
- 支持扩展编译器功能

#### 3.2.7 tools 模块

- **cmd**：命令行工具实现
- 提供 build、dev、init 等常见命令

#### 3.2.8 types 模块

- 定义项目中使用的所有类型
- 确保类型定义的一致性

### 3.3 tests 目录

- 所有测试代码必须放在 `/tests` 目录中，禁止在 `/src` 目录中编写测试
- 包含单元测试和集成测试
- 确保编译器功能的正确性
- 测试文件应按照功能模块进行组织

### 3.4 templates 目录

- 所有模板文件必须放在 `/templates` 目录中
- 模板文件应按照功能或类型进行组织
- 禁止在其他目录中存放模板文件

## 4. 代码规范

### 4.1 文档注释

所有 public 的结构体、枚举、方法、字段都需要文档注释，禁止使用后置注释。

### 4.2 代码风格

- 遵循 Rust 官方代码风格
- 使用 rustfmt 格式化代码
- 保持代码简洁明了

### 4.3 错误处理

- 使用 Result 类型处理错误
- 提供详细的错误信息
- 实现统一的错误处理机制

## 5. 项目维护规范

### 5.1 版本控制

- 遵循语义化版本规范
- 保持版本号的一致性

### 5.2 测试

- 确保所有测试通过
- 定期运行测试套件

### 5.3 文档

- 保持文档的更新
- 提供清晰的使用说明

## 6. 迁移指南

对于现有的模板引擎项目，需要按照以下步骤进行迁移：

1. 按照统一的目录结构重新组织代码
2. 将解析器替换为 oaks 提供的解析器
3. 复用 nargo 提供的组件
4. 确保所有测试通过
5. 更新文档

## 8. 结论

通过遵循以上规范，我们可以确保所有模板引擎编译器具有一致的架构和代码组织，提高代码的可维护性和可扩展性。同时，通过复用 nargo 和 oaks 提供的组件，我们可以减少代码冗余，提高开发效率。
